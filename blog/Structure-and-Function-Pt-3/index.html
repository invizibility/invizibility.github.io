Structure and Function Pt. 3: Building a Data Ontology

Once we did a bit of work establishing the blind spots of current data repositories hosting structure and function data, we worked to crystallize the principles that would inform our newly developed organizational paradigm for storing, browsing, and visualizing collections of structure and function data. These principles were inspired both by the desire to go beyond currently available organzitional schema, as well as the desire to constrain our problem into a manageable enough scope to make the development of a prototype interface viable.

Visual Brainstorming

As I did research about CHiP-seq, conformation capture, and the various range of cell types that might populate our prototypical library, I kept a journal where I kept note of potentially useful organizational heuristics and sketched out my mental model of data ontology. Gradually pulling a visual schema out of this organizational thinking exercise connects to some of the higher level goals of this project for me, namely using visualization to traverse high dimensional data-spaces. I specifically tinkered with how to create a visual model describing how structure data-space and function data space might interact with one another, and how the varying layers of complexity on each side might affect those interactions. Using this nested two-panel format to describe the data space and the challenges of navigating it ended up being a great tool for communicating with Noah about the mental models used to formulate structure-function hypotheses.

One crucial developmental point was coming to the conclusion that we needed ways to organize experiments both by what we know about them (e.g. attributes like phenotype, method, quality, date) as well as by direct computational comparison of experimental data which might capture similarity we don’t know about. It turns out many CHiP-seq experiments are done to capture binding affinities of proteins which have completely unknown function. If there is a protein with known function, and it has a similar binding pattern as a protein of unknown function, this a meaningful relationship we’d like to know about! Is there a way to make obvious such connections through a visual paradigm?

Organizing + Curating Structure Data

I tried to pin down actual quantitative measures of complexity that we would be dealing with when creating an interface to accomodate this ontology. One key constraint we decided on at this stage was to narrow our cell lines of interest to the 7 lines outlined in Rao et. al 2014, which is one of the most cited conformation capture experiments. The 7 cell lines for which Rao et. al 2014 has structural data are lines that have a plethora of functional experiments available (namely CHiP-seq), and furthermore the actual conformation capture done in this publication is regarded to be of very high quality. These data will be the inputs for researchers like Noah who experiment with various computational geometry techniques to generate polymer models of DNA, which constitute the actual structure files.

Techniques of generating 3 dimensional genomic structures from pairwise HI-C data and / or specific functional experiments have not converged at a consensus “best practice”. As such, there is a need to create a flexible organization scheme accounting for the fact that the majority of 3D structures are acquired directly from researchers experimenting with computational geometry rendering algorithms. Some algorithms render a single structure from a single Hi-C pairwise matrix; others render a population of structures using statistical methods across multiple Hi-C and functional reads. While each algorithm might be optimized for a specific use case by the researcher who designed it, no single algorithm can characteristize the provenance of 3 dimensional structure data in general. This is a stark contrast from the functional data, which we acquire from ENCODE after it has been standardized and quality controlled to a particular format. The biggest issue we see immediately on the structure side from a design perspective is to create a legible hierarchy that can grow and adapt as the techniques for rendering 3d structures become more nuanced and complete. This adaptability will allow structures generated from new algorithms to fit within our larger framework of ‘biologically meaningful’ structure-function matching.

Organizing + Curating Function Data

Because we are interested in structure-function matches, we applied the same 7 cell line constraint from the Rao et. al 2014 paper that we used on the function side. In terms of experiment type, we are initially only going to look at organizational methods for CHiP-seq, since that seems to be the type with the largest amount of variety for our 7 cell lines. The most fine grained data type on the function side is the peak file, which is basically a histogram describing a variable quantity across the length of a genome. On ENCODE, it turns out just about every CHiP-seq experiment has multiple peak files, most of which are ‘replicates’ which ensure the reads are repeatable for a given experiment. While there are some time-dimensional experiments for which multiple peak files are an interesting level of data variety, we are going to constrain ourselves to 1 peak file per experiment to manage our complexity. Same as the CHiP-seq experiment type constraint, once we build the architecture to organize this, we might be able to expand it to accomodate more.

Unifying Data Spaces

Bringing these two panels together as seen in the earlier notes helps us to illustrate the more nuanced conceptual problems of traversing current data repositories to observe interesting structure function relationships.

Noah referred earlier to an ideal interface being able to expose biologically meaningful matches between structure and function. Our visual model gives an idea of what types of complexity need to be wrangled in order to allow these matches. To continue our brainstorming we situated some of the challenges and complexities on the two panel model itself.

This framework for thinking about potential relationships between structure and function data allowed us to begin our iterative design process. At the heart of these issues was our actionable design question: what is the optimal way to pair a structure experiment and function experiment in an experiment browser that maximally facilitates exploration of the data space? The above issues constitute concrete obstacles to this exploration that we’d like to design around, and can largely be grouped into issues of data size, data heterogeneity, or clarity of metadata relationships. From here the next step was for me to dive into feature ideation to see what types of visualization methods we could leverage to get around these specific obstacles.
